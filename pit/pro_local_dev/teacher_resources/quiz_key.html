Professional Local Development — Quiz

ANSWER KEY — TEACHER USE ONLY
About this key: Student wording will vary. The answers below describe the concepts that should appear for full credit, drawn from the lesson page. Anything substantively equivalent counts. Watch for the listed misconceptions — those are the most common partial-credit traps.

1. Before writing any code in a new environment, what should a professional developer do first, and why?

Look for both ideas:
  • Validate / verify the environment (Python is installed and on PATH, terminal works, VS Code can run code).
  • Why: a bad PATH, missing interpreter, or wrong terminal will silently waste hours later. Validate first, then code — the same logic as a pilot's pre-flight checklist or a surgeon's time-out.
Full credit needs: both what to do and why. "Make sure Python works" alone is partial credit; tying it to "so I don't waste time debugging the environment instead of my code" earns full credit.

2. What does pip do, and what command did you use to install CMU Graphics on your local machine?

Look for:
  • pip is Python's package manager — it downloads and installs third-party Python libraries from the Python Package Index.
  • Command: pip install cmu-graphics
Accept: "installs packages" or "installs libraries." Don't accept "it runs Python code" or "it's a Python program" — those are wrong.

3. Name two differences between running CMU Graphics in the browser vs. locally in VS Code.

Any two of (or substantively similar):
  • Local runs require an explicit from cmu_graphics import * and cmu_graphics.run(); the browser hides this.
  • Local code has access to the real filesystem (load/save images, files, data); the browser is sandboxed.
  • Local code is faster and not throttled by the browser tab.
  • Local errors appear in the terminal; the browser shows them in a custom error panel.
  • Local projects can be put under Git version control; browser-only programs cannot.
  • Local code can be shared as a real .py file; browser programs live inside CMU Academy.
  • Local code requires the developer to manage dependencies (pip install); the browser includes everything automatically.
  • Local code is stricter about syntax and case; the browser hides some quirks.
Partial credit: one valid difference = half credit. "It's different" with no specifics = no credit.

4. Define repository, commit, and staging area.

All three required:
  • Repository: a folder Git is tracking; the history lives in the hidden .git directory.
  • Commit: a snapshot of the project at a moment in time, with a message describing the change.
  • Staging area: the "shopping cart" of changes selected to include in the next commit (added via git add).
Common error: Saying a commit is "saving" the file — saving writes to disk; committing records a snapshot in Git's history. They are different operations.

5. Git vs. GitHub.

Look for: Git is the version control software that runs locally on your machine. GitHub (or Gitea, GitLab, Bitbucket) is a hosting service — a website that stores Git repositories on a server so people can share them. The analogy from the lesson is Microsoft Word vs. OneDrive: Word is the program, OneDrive is the cloud where you store the files.
Don't accept: "GitHub is the newer version of Git" or "GitHub is owned by Git" — both are wrong.

6. Order the Git commands.

Correct order:
  1. git init
  2. git status  (verify what's tracked / staged)
  3. git add .
  4. git commit -m "Initial commit"
Scoring: All four correct = full credit. git status placement is flexible (it can also appear between add and commit) — accept either, as long as init is first, add comes before commit, and commit is last.

7. A good comment explains the ___ , not the ___ .

Answer: A good comment explains the WHY, not the WHAT.
Accept: "reason" / "intent" / "purpose" for WHY; "code" / "syntax" / "obvious operation" for WHAT.

8. An industry other than software that uses version control / revision tracking.

Acceptable examples (from the lesson):
  • Law — redline / Track Changes on contracts; numbered amendments.
  • Engineering / manufacturing — PLM systems tracking CAD revisions and Bills of Materials; configuration management.
  • Accounting / finance — SOX-compliant audit trails on financial records.
  • Healthcare — EHRs are append-only; chart notes get addenda, not edits.
  • Film / music — Pro Tools, Avid, Final Cut version snapshots.
  • Government — bill revisions; the U.S. Code as a change log.
  • Architecture — numbered/dated drawing revisions.
Other reasonable industry examples (e.g., Wikipedia revision history, scientific paper preprint versions) also accepted.
Full credit needs: industry + what is tracked. "Doctors use version control" alone = partial; "Hospital EHR systems keep every chart note as an addendum so the history is permanent" = full.

9. Why commit often with descriptive messages?

Look for any 2–3 of:
  • You can roll back to a specific point if something breaks — harder if everything is one giant commit.
  • Each commit message becomes a record of why a change was made (like a project changelog).
  • Easier to review and code-review small, focused changes.
  • Easier to find when a bug was introduced (e.g., git bisect).
  • Easier for collaborators to follow your reasoning.
  • Reduces the risk of losing large amounts of work to a single bad commit.
Partial credit: "so you don't lose work" alone is partial; tie it to rollback, review, or collaboration for full credit.

10. Identify a moment from today that exercised flexibility, open-mindedness, initiative, or willingness to learn.

Look for:
  • A specific moment from the lab (not generic "I learned a lot today").
  • An explicit link to one of the four named TEKS skills.
  • 2–3 sentences of substance.
Strong answers often involve: troubleshooting an error without asking for help, switching machines after one was unusable, helping a peer, adapting a CMU program that didn't work locally on the first try, or learning a brand-new tool (Git, terminal) without prior experience.
Scoring rubric:
  • 4–5 pts: Specific moment, named TEKS skill, clear reflection.
  • 2–3 pts: Generic moment or missing the named skill linkage.
  • 0–1 pt: No specific example, or pure restatement of the prompt.
