Pacing: This project spans two 180-minute class periods (360 minutes total). Class 1 focuses on building, publishing, and the first round of peer review. Class 2 focuses on Git history surgery, an open experimentation lab where you pick your own Git adventure, and submitting your deliverable. The PDF is due at the end of Class 2.

Learning Objectives

By the end of these two classes, you will be able to:

  1. Create a Gitea account and a public repo named my_first_repo_lastname.
  2. Build a small HTML / CSS / JS splash page (AI assistance permitted) and push it to the classroom server.
  3. View your work live at a public URL hosted on the classroom Pi.
  4. Exchange specific, actionable feedback with another student and drive a real iteration loop, committing each change you like.
  5. Discard uncommitted changes safely with git restore.
  6. Roll back a specific past commit with git revert and rebuild state with git cherry-pick.
  7. Pick two or more experiments from the Open Experimentation Lab (branches, merge conflicts, stash, reflog, blame, two-machine simulation, or self-proposed) and document what you learned.
  8. Submit a deliverable with screenshots, peer-feedback notes, and a written reflection.

Why This Matters Beyond CS

Today's practice — ship something, hear feedback, fix it, ship again, and never overwrite the past so you can always roll back — is the same instinct used by film editors saving versioned cuts, surgeons running pre-op timeouts, auditors keeping append-only ledgers, and SRE teams on call at $200K+ jobs. Self-hosted infrastructure (the Raspberry Pi serving this lesson) is a real, paid IT specialty in hospitals, defense contractors, banks, and small businesses that don't trust their data to public clouds.

Class Expectations

  • Phones away — we do not need them (ask if you believe there is an exception).
  • No food. Liquids in sealed containers only.
  • Be respectful when giving and receiving peer feedback. Specific is kind; vague is not.

Lesson Expectations

  • Capture exact error text on any failure. Documented failures are worth full credit; pretending something worked when it didn’t is academic dishonesty.
  • Read git status and git diff before staging or discarding — this is professional muscle memory.
  • Never commit secrets, credentials, or files from a OneDrive-synced folder.
  • Try at least one documented workaround before escalating — bring the exact error text with you when you do.
