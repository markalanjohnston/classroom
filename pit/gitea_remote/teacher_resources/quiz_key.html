Gitea Workflow & History Surgery — Quiz

ANSWER KEY — TEACHER USE ONLY
About this key: Student wording will vary. The answers below describe the concepts that should appear for full credit, drawn from the lesson page. Anything substantively equivalent counts. Watch for the listed misconceptions — those are the most common partial-credit traps.

1. Local vs. remote repository.

Look for:
  • Local repo — lives on the student's own machine (their laptop / lab PC).
  • Remote repo — a copy hosted on a server (in this class, the Pi at 10.12.6.236) that other people can also access.
  • Both contain the full history. The remote exists so collaborators can sync and so the work survives a dead laptop.
Common error: Saying "remote is read-only" or "remote is just the latest file" — both wrong. The remote has the full history too.

2. What does git push do? What would the partner see if you committed but didn't push?

Both ideas required:
  • git push uploads local commits to the remote so the remote (and therefore the public URL) is up to date.
  • If you only committed and didn't push, the partner sees whatever was on the remote before — the empty-repo page on a brand-new repo, or the previous version of the splash page on an existing one. They do not see the new commit.
Full credit needs: both the definition of push AND the realization that commits are local until pushed. "Push sends it to GitHub" alone is partial; tying it to "the partner can't see the change yet" is full.

3. Fill in the blanks.

Answers:
  • a. Ctrl+S writes / saves the file to disk.
  • b. git commit takes a snapshot of the staged changes.
  • c. git push sends commits to the remote.
Accept: "stores"/"writes" for (a); "checkpoint"/"point in time"/"version" for (b); "Gitea server"/"origin"/"server" for (c).

4. Order the commands for shipping a peer-feedback fix.

Correct order:
  1. git status  (see what's modified)
  2. git add index.html
  3. git commit -m "Fix low-contrast headings"
  4. git push
Scoring: All four correct = full credit. git status placement is flexible (it can also appear after add) — accept either, as long as add precedes commit and push is last.

5. git restore vs. git revert.

Look for both ideas:
  • git restore <file> throws away uncommitted edits in the working tree. Used before you've committed.
    Scenario: "I made a bunch of changes I hate; I want to go back to the last commit's version of the file."
  • git revert <hash> undoes a specific past commit by adding a new commit that cancels it out — safe even after pushing.
    Scenario: "We pushed a commit that broke the site. Roll it back without rewriting history that other teammates already have."
Full credit needs: both definitions and a scenario for each. Common error: confusing them or saying "revert" works on uncommitted changes (it doesn't).

6. Bad commit pushed; teammates depend on later commits. Which command, and why not the others?

Look for:
  • git revert <hash-of-bad-commit> (then git push).
  • Why: revert adds a new commit that undoes only the bad one, leaving every other commit intact and keeping history shareable.
  • Why not git reset --hard + force push? It rewrites history that teammates already have on their machines — causes chaos for everyone who pulls.
  • Why not git restore? It only handles uncommitted changes — doesn't undo a commit at all.
Full credit needs: naming revert AND explaining why at least one other option is wrong. "Use revert" alone = partial.

7. When would you use git cherry-pick?

Look for a specific scenario like:
  • "A teammate wrote one tiny bug fix on a different branch among 12 unfinished commits. I want only that one fix, not the rest."
  • "A hotfix needs to go onto the stable release branch — I cherry-pick just that commit from main."
  • "I reverted a feature, then later decided I wanted it back — I cherry-picked the original commit to reapply its changes."
Key idea: cherry-pick = take one specific commit's diff and reapply it on top of where you are now, without bringing along the surrounding commits.
Partial credit: Generic "use it to pick a commit" — partial. Specific realistic scenario with the "just one commit, not the rest" idea — full.

8. Why must the repository be Public for the lesson?

Look for: The classroom nginx web server is intentionally configured to serve files only from public repos. A private repo would still clone, push, and commit fine, but its files would not be served at http://10.12.6.236/sites/username/reponame/. The Public setting is what makes the splash page reachable by a peer.
Don't accept: "Private repos cost money" or "Git doesn't work on private repos" — both are wrong. The mechanism is the nginx config, not Git itself.

9. Best commit message — and why the others are worse.

Best answer: (c) Increase header contrast for readability
  • (a) stuff — meaningless; tells future-you nothing.
  • (b) final FINAL fixed for real this time — emotional / non-specific; doesn't say what changed.
  • (d) the long version — rambling, buries the signal in noise, mentions a name (Maria) that's irrelevant for the changelog. Worse than (a) because it disguises uselessness as effort.
Good messages are imperative (Fix, Add, Update, not Fixed/Adding), specific, and ~50 characters for the summary line.
Full credit: Correct choice AND explanation referencing imperative mood, specificity, or brevity. Just circling (c) = partial.

10. "Never overwrite the past" outside of programming.

Acceptable examples (from the lesson):
  • Healthcare / EHR — chart notes are append-only; you add an addendum, not edit the original.
  • Law — contracts are amended through numbered, dated revisions; redlines are tracked.
  • Accounting / finance — SOX-compliant audit trails on every change.
  • Manufacturing / engineering — PLM systems and configuration management track CAD and BOM revisions.
  • Film / music — Pro Tools, Avid, Final Cut snapshot versions of cuts.
  • Government — numbered bill revisions; U.S. Code as a permanent change log.
  • Photography — non-destructive editing in Lightroom; the original raw is never overwritten.
Other reasonable industries (Wikipedia revision history, scientific preprints, etc.) also accepted.
Full credit needs: Specific industry + what is tracked + tie to the "never overwrite / can roll back" idea. "Doctors track stuff" alone = partial; "Hospital EHRs are append-only by law — you can never edit a chart note, only add an addendum, so the history is permanent" = full.
