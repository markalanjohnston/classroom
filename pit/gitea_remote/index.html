Publishing to the Classroom Gitea Server

Required extension to the Pro Local Dev practicum — push your first website to a real, self-hosted Git server running on the classroom Raspberry Pi

When to do this: This is a required extension of the Pro Local Dev practicum — not optional. If you finished today's main lab early, start it now. Otherwise, you will complete it during the next class period. The screenshot from Step 7 becomes part of the same PDF deliverable.

Fun fact: The Gitea server, web hosting (nginx), and the public file viewer you'll use today are all running on a single Raspberry Pi sitting on Mr. Johnston's desk. No cloud, no AWS, no Microsoft — just a credit-card-sized computer and good open-source software. If a Pi can host this, you can host things too.

What You'll Do

  1. Create an account on the classroom Gitea server.
  2. Create your first remote repository.
  3. Clone it to your local machine with Git Bash.
  4. Build a simple index.html splash page (you may use the AI of your choice).
  5. Configure your Git identity (name + email).
  6. Stage, commit, and push — the new Git command this lesson is built around.
  7. View your file publicly hosted at the classroom web address.

New term today: a remote is a copy of your repository that lives on a server somewhere — in this case, on the Pi.

1 Create your Gitea account

  1. Open a browser and go to http://10.12.6.236:3000/.
  2. Click Register.
  3. Pick a username you can live with publicly (it will appear in URLs and on your repo page). Use your school email if you want password reset to actually work.
  4. Sign in.
The server is only reachable from inside the classroom network. If you try this on your phone using cellular data, it will fail. That's not a bug — it's intentional. Self-hosted services live where the hardware lives.

2 Create a repository named test

  1. In the top-right of Gitea, click the + icon, then New Repository.
  2. Name it exactly: test
  3. Leave Initialize Repository unchecked. We want an empty remote so we can practice the full push workflow.
  4. Click Create Repository.

Gitea will show you a "quick setup" page with a URL that looks like:

http://10.12.6.236:3000/your-username/test.git

Copy that URL — you'll need it in the next step.

3 Clone the empty remote to your local machine

DO NOT clone into a OneDrive folder. Cloud-synced folders (OneDrive, Google Drive, Dropbox) constantly rewrite files in the background. This collides with Git's .git directory and causes file-lock errors, missing objects, and corrupted repos. Use a local folder that is NOT synced to OneDrive.
  1. Open Git Bash (Start menu → search "Git Bash").
  2. Move to your local Documents folder: 
    cd ~/Documents
pwd
  3. Read the pwd output carefully. If it shows something like /c/Users/you/OneDrive/Documents — that's bad. You need a non-OneDrive path. Try one of these instead: 
    # Option A: use the user profile root
cd /c/Users/$USER

# Option B: make a dedicated PIT projects folder
mkdir -p /c/Users/$USER/pit-projects
cd /c/Users/$USER/pit-projects
    When you run pwd again, the path should not contain the word "OneDrive". If you can't find a non-OneDrive location, raise your hand.
  4. Clone the empty remote (replace your-username): 
    git clone http://10.12.6.236:3000/your-username/test.git
cd test
Expected message: Git will warn that you appear to have cloned an empty repository. That is exactly what we wanted — we created the remote first specifically so we'd have something real to push to. The clone wired up the connection between this local folder and the remote on the Pi.

Confirm the remote is wired up:

git remote -v

You should see origin pointing at the Pi's URL.

4 Build an index.html splash page

  1. Open the test folder in VS Code: 
    code .
    (If code isn't recognized, just open VS Code and use File → Open Folder.)
  2. Create a new file named exactly index.html.
  3. Use the AI of your choice (ChatGPT, Claude, Gemini, Copilot) to generate a one-page personal splash. Suggested prompt: 
    Generate a single-file index.html splash page for me.
Include: my name as a heading, a short bio paragraph,
3 things I'm working on this semester, and inline CSS
to make it look clean. No external links or scripts.
  4. Paste the generated HTML into index.html and save (Ctrl+S).
  5. Open the file in your browser to make sure it renders before you push.
Read what the AI gave you before you paste. If it included external scripts, tracker pixels, or links to sites you don't recognize, strip them out. You're about to publish this publicly.

5 Configure your Git identity

Git needs to know who is making commits. Set this once (the --global flag writes to your user profile, so it carries across future repos):

git config --global user.name "Your Name"
git config --global user.email "you@yourschool.edu"

Use the same email as your Gitea account so your commits get linked to your profile on the server.

Verify it:

git config --global --list
Why "global" is OK in this lab: on a Windows lab machine, --global writes to your personal user profile, not to a machine-wide location. The next student to log in with their own account won't see your config.

6 Stage, commit, and push

Two of these commands you already know from the Pro Local Dev lesson. The third — git push — is the new one. Push sends your local commits up to the remote on the Pi.

git status
git add .
git commit -m "Add personal splash page"
git push
First-time push prompt: Git Credential Manager will likely open a browser tab pointed at the Gitea login. Click Authorize (or sign in if asked), then return to Git Bash. The push will complete. After this first time, your machine will remember the credentials.
Checkpoint: The last lines of git push output should show something like main -> main and a couple of object counts. Screenshot it.

7 See your file on the remote — and live on the web

  1. Refresh your Gitea repo page at http://10.12.6.236:3000/. You should now see index.html listed instead of the "empty repository" message.
  2. Now for the cool part — the classroom server also serves your files publicly through nginx, similar to GitHub Pages. Visit: 
    http://10.12.6.236/your-username/test/
    Your splash page should load in the browser, hosted off the Pi.
Final checkpoint: Screenshot of your publicly hosted splash page rendered from http://10.12.6.236/your-username/test/. Add this screenshot to your PDF deliverable from the Pro Local Dev lab as a new "Remote Hosting & Push" section.

Career "Self-hosted" is a real, paid skill

You just stood up your own Git server, made an account, pushed code, and viewed it on a public URL — on hardware running in the same room. That's not a toy. System administrators, DevOps engineers, cloud architects, and IT operations specialists do exactly this for a living, at scale, in industries that don't trust their data to third-party clouds:

  • Hospitals & clinics — HIPAA-regulated data often runs on on-premises servers, not cloud.
  • Defense contractors — classified work runs on private "air-gapped" networks.
  • Banks & insurance — core transaction systems are still self-hosted for compliance and audit reasons.
  • Universities & research labs — researchers host their own Git, datasets, and compute clusters.
  • Small businesses — many local IT shops install on-prem servers for cost or privacy reasons.

"Can run their own infrastructure" is one of the highest-paid skill clusters in IT. Today's Raspberry Pi exercise is the same shape as a $150K sysadmin job — just at one-thousandth the scale.

If Something Doesn't Work

  • "fatal: not a git repository" — you're not inside the test folder. cd test and try again.
  • "Permission denied" / "authentication failed" — your Git identity email doesn't match your Gitea account, or you canceled the browser authorize step. Re-run git push and complete the browser prompt.
  • "index.lock" / "unable to write" errors — you're inside a OneDrive folder. Move the repo to a non-synced location (see Step 3) and clone again.
  • Can't reach 10.12.6.236 — you're probably on cellular data or the guest network. Connect to the classroom Wi-Fi.
  • Public URL shows 404 or directory listing — the file might be named Index.html (capital I) or saved with a hidden .txt extension. Check the actual filename in Git Bash with ls.
  • Page loads but is blank — the AI may have given you incomplete HTML. View source in the browser; you should see at minimum <html>, <head>, <body> tags.

Document any error you hit — exact text, what you tried, what fixed it. That's the report you'd submit on the job.
