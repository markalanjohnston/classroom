0 Start Here — What You're Building

⏱️ About 10 minutes to read through

You've spent a whole semester writing Python programs in CMU CS Academy — drawing shapes, responding to mouse clicks, tracking state with variables. In this project, you'll take one of those programs and turn it into a real mobile app that runs on your phone or tablet.

You will NOT be writing JavaScript by hand. Instead, you'll use Claude.ai (an AI assistant) to refactor your Python code into mobile code, then fix errors together until the app runs.

Here's what the finished thing looks like

The example project — a Mickey Mouse app where tapping each part reveals a colorful version underneath.

What you're going to do

  • Pick a CMU CS Academy project you've already written (or use the example provided)
  • Copy it into Claude.ai and ask it to rebuild the project as a mobile app
  • Paste the new code into the CodeHS sandbox
  • When the code has errors (it will!), copy the error back to Claude and keep going
  • Once it runs, open it on your phone or tablet using the Expo Go app
  • Write a short reflection about what transferred and what didn't

The real lesson here

The point is not to learn JavaScript. The point is to see that the same programming concepts you've been using all semester — variables, events, conditionals, drawing shapes — exist in every language and on every platform. The words change. The ideas don't.

Warning: the AI will make mistakes

Claude will not get it right on the first try. When I built the example project, it took 6 rounds of fixing errors. That's normal. Your job is to read each error, describe it to Claude, and keep going. This is how real programmers work with AI.

Start Step 1

1 Pick the CMU Project You'll Refactor

⏱️ 10 minutes

You have two options. Pick whichever one sounds better to you.

Option A — Use your own project (recommended)

If you've built a creative task or mini-project in CMU CS Academy that you're proud of and that uses shapes and mouse clicks, use it. This will make the project way more interesting to you.

A good project to pick has:

  • Several shapes (ovals, rectangles, circles, polygons)
  • An onMousePress handler that does something when you click
  • Some kind of change — shapes appearing, disappearing, moving, or changing color

Avoid projects that use onStep animations, loops, or lots of advanced features — those will make debugging much harder. Start simple.

Option B — Use the example Mickey Mouse project

If you don't have a good project to refactor, use the example. Download the starter:

  • Download the starter file: cmu-starter.py
  • Open CMU CS Academy's IDE and paste the code in
  • Fill in the onMousePress handler at the bottom. The solution is redacted — you need to write logic that checks if the click hit any retro part and clears it. Hints are in the file.
  • Run it and verify: clicking each black-and-white part should remove it and reveal the colorful version underneath

Hint for the Mickey solution

Loop through retroMickey.children. For each part, check .hits(mouseX, mouseY) and call .clear() if it was hit.

Before moving on — make sure:

  • You have a CMU CS Academy project that actually runs
  • Clicking the canvas does something visible
  • You can copy the full Python code to your clipboard

3 Ask Claude to Rebuild It as a Mobile App

⏱️ 5 minutes

This is where the magic starts. You're going to copy your CMU Python code and paste it into Claude with a specific prompt.

  • Copy all of your Python code from CMU CS Academy (Ctrl+A then Ctrl+C, or Cmd+A then Cmd+C on Mac)
  • Open Claude.ai and start a new chat
  • Paste this prompt, then paste your code at the end where it says [paste your code here]:
Please refactor this CMU CS Academy Python project into a React Native mobile app that runs in the CodeHS sandbox. Important constraints for CodeHS: - The root component MUST be called "App" - Use ES5 class syntax with this.state and this.setState (CodeHS does NOT support React hooks) - Do NOT use react-native-svg (not available). Use core <View> with borderRadius for circles/ovals. - Include AppRegistry.registerComponent("App", () => App) at the bottom - Include BOTH: module.exports = App AND module.exports.default = App - Use onLayout (not Dimensions.get) so the layout works in both the CodeHS preview AND a real phone Here is my Python code: [paste your code here]

Why this prompt is specific

The constraints in this prompt come from 6 rounds of debugging I already did for you. Including them upfront saves Claude (and you) a lot of back-and-forth. If you skip them, you'll just hit those same errors yourself.

  • Wait for Claude to respond. It will output a big block of JavaScript/JSX code.
  • Copy Claude's code (not the Python, the JSX it just wrote). Click the copy icon on Claude's code block, or select it all and copy.

Before moving on — make sure:

  • Claude has given you a big block of React Native code
  • You can see class App extends React.Component somewhere in the code
  • You've copied the code to your clipboard

4 Paste Into CodeHS and Run It

⏱️ 5 minutes

  • Go to your CodeHS sandbox (the React Native one you made in Step 2)
  • Delete all the starter code that's already there (Ctrl+A, Delete)
  • Paste Claude's code
  • Click the Run button (usually a ▶ play icon at the top of the editor)

CodeHS sandbox — your code goes on the left, the preview shows on the right.

One of three things will happen

  • It works on the first try. Congratulations, you won the lottery. Skip to Step 6.
  • You see an error message. Expected! Go to Step 5.
  • The preview is blank or looks wrong. Also go to Step 5 — treat "nothing happens" as an error.

    5 Debug With Claude (the Main Skill)

    ⏱️ 20–40 minutes (most of the project happens here)

    This step is the real skill of the whole project. You're going to have a conversation with Claude where you describe problems and it suggests fixes.

    The debugging loop

    Repeat this until the app works:

    • Read the exact error message in CodeHS. Copy the full error — don't paraphrase.
    • Go back to the Claude chat (same conversation, not a new one)
    • Paste the error like this:
      I pasted your code into CodeHS and got this exact error: [paste error message here] Can you fix it?
    • Copy the updated code Claude gives you
    • Replace everything in CodeHS with the new code
    • Run it again

    Include the WHOLE error message, word for word

    "It didn't work" is not enough. "Here is the exact error: TypeError: _react2.default.useState is not a function (240)" gives Claude something to actually work with.

    Errors you might see (and roughly what they mean)

    These are the six I hit when I built the example. Yours may be different — that's fine.

    "Looks like you forgot to import AppRegistry"

    Missing the line that tells CodeHS what to run. Ask Claude to add AppRegistry.registerComponent.

    ReferenceError: App is not defined

    The main component is called something else (like MickeyScreen). Ask Claude to rename it to App.

    TypeError: _react2.default.useState is not a function

    CodeHS doesn't support modern React hooks. Ask Claude to rewrite using ES5 class syntax with this.state.

    Invariant Violation: Minified React error #130

    Code is using react-native-svg, which isn't available. Ask Claude to use <View> with borderRadius instead.

    "No default export of 'app.js' to render"

    Works in the CodeHS preview but not on a real phone. Ask Claude to add module.exports.default = App.

    Looks right on phone, wrong in CodeHS preview (or vice versa)

    Hard-coded screen sizes. Ask Claude to use onLayout instead of Dimensions.get.

    If you're really stuck after 8+ rounds

    Try starting a fresh Claude chat with your ORIGINAL Python code and an even more specific prompt. Sometimes the conversation gets tangled and a clean restart helps. You can also compare against the known-working example to see what the final code should roughly look like.

    You're done with this step when:

    • No error messages appear in CodeHS
    • You can see something in the preview pane
    • If your project was interactive (clicking, tapping, etc.), that interaction works in the preview
    Back Next: Test on Phone

    6 Run Your App on Your Phone

    ⏱️ 10 minutes

    Now for the best part. You've built a mobile app — let's put it on an actual mobile device.

    • Install Expo Go on your phone or tablet if you didn't already (App Store for iPhone/iPad, Google Play for Android)
    • In CodeHS, click the "Share" button (usually near the top). This gives you a shareable link.
    • On your phone, open Expo Go, then either scan the QR code or paste the CodeHS link
    • Wait for it to load (first load can take 30+ seconds)
    • Test the interaction — tap, click, whatever your project does. It should work the same as in the CodeHS preview.

    Your app running on a real phone via Expo Go.

    Phone and computer must be on the same Wi-Fi

    Expo Go loads your code from CodeHS over the network. If your phone is on cell data and your computer is on school Wi-Fi, they may not connect. Put your phone on the school Wi-Fi.

    Works in CodeHS preview but not on phone?

    This was "Round 5" and "Round 6" in the debugging log. Go back to Step 5 and tell Claude: "It works in the CodeHS preview but not in Expo Go on my phone."

    You're done with this step when:

    • You can see your app on your phone screen
    • The interaction (tapping, etc.) works on the phone
    • Take a screenshot of the app running on your phone — you'll need it for Step 7
    Back Final Step: Reflect & Submit

    7 Reflect & Submit

    ⏱️ 15–20 minutes

    The coding is done. Now the most important part: thinking about what you just did.

    Answer these four questions

    Write at least a few sentences for each. Answer honestly — "the AI got it right every time" is not a believable answer.

    1. Which concepts transferred directly from Python/CMU Graphics to React Native?

    Think about things like: variables, if statements, event handlers, drawing order, state.

    2. Which concepts required a workaround — something that didn't translate cleanly?

    For example: mouse events → touch events, or Ovals → Views with borderRadius.

    3. What did Claude get WRONG on the first try, and how did you know it was wrong?

    Describe at least one specific error, what the error message said, and how you figured out what to ask Claude to fix it.

    4. What decision did YOU make during this project that Claude could not have made?

    What did you contribute that an AI can't? This is the most important question.

    What to submit

    • Screenshot of your app running on your phone (from Step 6)
    • CodeHS Share link to your finished sandbox
    • Your answers to the four reflection questions above

    Submit through the usual class system. Check with Mr. Johnston if you're not sure which one.

    Bonus challenge (optional)

    Change something about your mobile app that makes it better on a phone than on a desktop. Examples: use the phone's full screen, add haptic feedback on tap, change the layout for portrait vs landscape, add a cool-looking title bar. Describe what you did in your reflection.

    You're done!

    • You took a Python project and turned it into a mobile app
    • You worked with an AI through an actual debugging process
    • You saw that programming concepts translate across languages
    • You shipped something that runs on a real device

    That's a real engineering workflow. Nice work.
