Minecraft Unit · Final Project

Independent Capstone Project

Design Your Own Build. Show Everything You Learned.

This is a 50-point project — double a normal lesson. The rubric is stricter, the reflection is longer, and you are expected to work independently. Read this whole page before you start coding.

The Big Idea

Lessons 1–5 gave you tools: Agent movement, events, loops, the Builder, relative coordinates, blocks.fill(), mobs.spawn(), iteration variables used as coordinates, modulo, distance math, and parameterized helper functions.

Your job now is to design and build something of your own — a project that pulls those tools together into one coordinated program. You may borrow code from any previous lesson, but you must transform it. Copy-paste with the numbers unchanged is not a project; it's a screenshot of someone else's work.

Time & Working Conditions

  • In class: One 90-minute block. Use the whole time.
  • At home: You may continue on your district-issued laptop. MakeCode runs in the browser; no Minecraft install is required to edit code, but you'll need Minecraft to take the build screenshot.
  • Independence: The code you submit must be your work. You may ask for help debugging a specific error, but you may not have someone else write your build for you.

Project Requirements (all required)

1. An event entry point. Your program starts from a chat command (player.on_chat) or item interaction (player.on_item_interacted).

2. At least three concepts from Lessons 1–5. Pick from this list — you must use at least three:

  • Agent movement and placement (Lesson 1)
  • A for loop (Lesson 2)
  • The Builder or relative coordinates with ~ (Lesson 3)
  • blocks.fill() with an intentional FillOperation choice (Lesson 4)
  • mobs.spawn() with a loop and randpos() (Lesson 4)
  • Using i or j as a coordinate or in math (Lesson 5)
  • Modulo (%) for a repeating pattern (Lesson 5)
  • Distance math — abs/max or Pythagorean (Lesson 5)

3. At least one parameterized helper function that you actually call — def my_helper(cx, cz, size, ...): with the parameters used inside the body.

4. A coherent build. The result should be recognizable as a thing — a base, a temple, a stadium, a maze, an aquarium, a village, a vehicle, a sculpture. Not a random scatter of blocks.

5. Scale. Your build covers at least a 20×20 area or is at least 10 blocks tall.

The "Reuse + Transform" Rule

You are allowed to start from code in any prior lesson, including the Step 8 castle. You are not allowed to submit that code unchanged. If you reuse any helper, function, or block of code from a lesson, you must change it in at least three of the following five ways, and you must name the changes in your reflection:

  1. Different theme — not the same medieval castle / aquarium / staircase. Reskin it as something else.
  2. Different materials — swap at least two block types for different ones.
  3. Different shape or layout — change radius, height, number of pieces, or how they're arranged.
  4. Add at least one new helper the original lesson did not include.
  5. Add a behavior the original did not have — mobs, a modulo pattern, a list of materials, a second chat command, etc.

Submitting the Lesson 5 castle with one number changed will receive partial credit at best. Be obvious about your changes — we should be able to look at your code and your reflection side-by-side and see what's new.

What to Turn In

Submit one PDF file using the Submit Assignment button on this page. Build the PDF in Microsoft Word or Google Docs, then export it as a PDF before uploading.

Your document must include all four of the following, in this order:

  1. Project title and one-sentence description at the top of the page (e.g., "Sky Temple — a stepped pyramid surrounded by four obelisks with a fire pit on top").
  2. Screenshot(s) of your code in MakeCode. Every helper function's def line and the chat-command function must be readable. Multiple screenshots are fine if the code is tall.
  3. Screenshot of the result in Minecraft — the full build, with the player standing in it or an aerial shot for scale.
  4. Written reflection — answer all six questions below in 2–3 sentences each. The reflection is the biggest single part of your grade.

Upload only the PDF. Do not submit the Word doc, the Google Docs link, or the raw image files.

Reflection Questions

Answer all six. Aim for 2–3 sentences each. Use vocabulary from the lessons (iteration variable, parameter, composition, modulo, FillOperation, opposite corners, etc.).

  1. What did you build? Name the project and walk a classmate through what they would see if they ran it.
  2. Which three concepts from Lessons 1–5 did you use? For each one, point to the line or function in your code where it appears.
  3. What did you reuse from a previous lesson, and what changes did you make? Name at least three specific changes (see the Reuse + Transform rule above). If you wrote everything from scratch, say that and explain why.
  4. Walk us through your most interesting helper function. Explain each parameter — what does it control? What would happen if you doubled one of them?
  5. What was the trickiest bug or design decision, and how did you resolve it? Specific is better than general — "the loop was off by one because I used range(10) instead of range(11)" beats "I had bugs."
  6. If you had another 90 minutes, what would you add or improve? One concrete next step.

Strategy & Tips

  • Plan before you code. Spend the first 5–10 minutes sketching what you want to build. What helpers will you need? What shared center will they use?
  • Start from working code. Open a Lesson 5 helper you already understand, paste it in, and then transform it. Building from scratch in 90 minutes is harder than remixing.
  • Test as you go. Run your program after every new helper. Don't write 100 lines then try to debug them all at once.
  • Use full-screen screenshots for code clarity — Win+Shift+S.
  • Name things meaningfully. def watchtower(cx, cz, height): is better than def thing(a, b, c): — and it makes your reflection easier to write.
  • Write the reflection while it's fresh. The debugging story is much harder to remember tomorrow.

Stretch Goals (for the ambitious)

  • Compose four or more helpers at one shared center.
  • Use a list of materials indexed by i % len(materials) for varied patterns.
  • Add a second chat command that modifies the first build — light it up, fill it with mobs, demolish it.
  • Use both modulo and distance math in different parts of the program.
  • Add # comments above each helper explaining what it does.
  • Use the Agent (Lesson 1) and blocks.fill() (Lesson 4) in the same project — tools from opposite ends of the unit, working together.

Academic Integrity

This project is graded as your work. You may discuss ideas with classmates and you may ask for help on a specific error message. You may not have a classmate, a sibling, or an AI assistant write your build, your helpers, or your reflection for you. Submitting work that isn't yours — or that you can't explain in the reflection — will receive a zero and a parent contact.
