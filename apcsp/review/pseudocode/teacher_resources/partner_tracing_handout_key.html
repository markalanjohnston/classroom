AP Computer Science Principles

Partner Tracing Practice — Reading Pseudocode

ANSWER KEY — TEACHER USE ONLY
About this key: Trace tables show the values after each numbered loop pass (i.e., after the variable update at the bottom of the loop body has run). Students who write values during the iteration instead of after can still be correct — check that the sequence of branch decisions matches. The most common errors and partial-credit notes are flagged in yellow.

Snippet 1 — Loop with an IF / ELSE inside

a ← 1
b ← 10
REPEAT 4 TIMES
{
    IF (a MOD 2 = 1)
    {
        b ← b + a
    }
    ELSE
    {
        b ← b - a
    }
    a ← a + 1
}
DISPLAY(b)
Step a b Notes (which branch ran? what changed?)
Start 1 10 Initial values before the loop begins.
After loop 1 2 11 a was 1; 1 MOD 2 = 1 (odd) → IF ran: b = 10 + 1 = 11. Then a ← a + 1 = 2.
After loop 2 3 9 a was 2; 2 MOD 2 = 0 (even) → ELSE ran: b = 11 − 2 = 9. Then a ← 3.
After loop 3 4 12 a was 3; 3 MOD 2 = 1 (odd) → IF ran: b = 9 + 3 = 12. Then a ← 4.
After loop 4 5 8 a was 4; 4 MOD 2 = 0 (even) → ELSE ran: b = 12 − 4 = 8. Then a ← 5. Loop body has now run 4 times — exit.
Final output of DISPLAY(b) 8
Common errors
  • Running the loop 5 times instead of 4 (off-by-one on REPEAT count).
  • Using a MOD 2 as "is a even?" backwards — remember a MOD 2 = 1 means odd, so the IF fires on odd values of a.
  • Updating a before the IF/ELSE check instead of after. This flips every branch decision and gives a final b of 12.
  • Confusing MOD (remainder) with division. 3 MOD 2 = 1, not 1.5.

Snippet 2 — List with REPEAT UNTIL (watch the 1-indexing!)

nums ← [4, 7, 2, 9, 5]
total ← 0
i ← 1
REPEAT UNTIL (i > LENGTH(nums))
{
    IF (nums[i] > 3)
    {
        total ← total + nums[i]
    }
    i ← i + 1
}
DISPLAY(total)
Step i nums[i] total Did the IF fire? (yes / no, why?)
Start 1 — 0 Initial values. LENGTH(nums) = 5.
Pass 1 2 4 4 Yes — 4 > 3, so total = 0 + 4 = 4. Then i ← 2.
Pass 2 3 7 11 Yes — 7 > 3, so total = 4 + 7 = 11. Then i ← 3.
Pass 3 4 2 11 No — 2 is not > 3, so total is unchanged. Then i ← 4.
Pass 4 5 9 20 Yes — 9 > 3, so total = 11 + 9 = 20. Then i ← 5.
Pass 5 6 5 25 Yes — 5 > 3, so total = 20 + 5 = 25. Then i ← 6.
Stop check 6 — 25 6 > 5 is true, so the loop exits without running the body again.
Final output of DISPLAY(total) 25
Common errors
  • Treating nums[1] as 7 (the second element). AP CSP lists are 1-indexed, so nums[1] = 4.
  • Stopping the loop after Pass 4 because they read REPEAT UNTIL (i > LENGTH(nums)) as i ≥ LENGTH(nums). The condition is strict > — the body must run while i is exactly 5.
  • Adding 2 to total on Pass 3 (forgetting the IF gate).
  • Final answer of 27 = student added 2 anyway. Final answer of 20 = student stopped one pass too early.
Partner-check trap question — i ← 0 If i starts at 0, the very first iteration tries to read nums[0]. In AP CSP pseudocode, lists are 1-indexed, so index 0 is not a valid position — this is an out-of-bounds access (an index error). The trace breaks on Pass 1 before any addition happens.

Accept also: a student who reasons that the loop would also run one extra time at the end (because i reaches 5 one pass later) is showing good thinking — but the headline answer is that nums[0] is invalid, so execution fails before that ever matters.
